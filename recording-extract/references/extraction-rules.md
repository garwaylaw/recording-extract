# 萃取规则

## 输入范围

本 Skill 只处理文字类录音资料：

- 飞书妙记【文字记录】
- 飞书文档
- Word 文档
- TXT 或 Markdown 转写稿
- 用户直接粘贴的转写文本

本 Skill 不处理音频或视频转写。

## 规则维护

当用户新增可复用的萃取要求时，应更新 Skill 文件：

- 萃取逻辑、内容分类规则、资料处理规则、输出格式规则，维护在本文件。
- 可复用的误词校正和知识库检索规则，维护在 `keyword-corrections.md`。
- 单份文档的一次性要求不要写入 Skill，除非用户明确要求变成通用规则。
- 保持规则适合同事通过 GitHub 复用。优先使用可配置示例，不写死个人本地路径。

## 内容类型判断

萃取前先判断转写稿类型。

### 公司内部问题讨论会

优先记录：

- 问题本身
- 问题背景和原因
- 约束条件和分歧
- 改善方向
- 原文明确提到的后续行动、责任人和时间节点

### 公司内部复盘或总结会

优先记录：

- 复盘事实和数据
- 复盘结论
- 经验教训
- 改善计划
- 可复制的方法或案例

### 与外部人员的交流

优先记录：

- 市场信息
- 客户反馈
- 竞品信息
- 合作诉求
- 业务机会
- 对产品、销售、服务、运营或经营管理的建议

## 萃取风格

- 忠于转写稿原意。
- 重点内容不要过度概括。
- 不添加无依据推测。
- 可以调整表达顺序和结构，但不能改变原意。
- 删除明显重复和低价值闲聊。
- 保留有助于理解事项重要性的上下文。

## 数值处理

重要数值应放在对应上下文中保留，包括：

- 金额
- 价格
- 数量
- 日期和时间
- 比例
- 目标值
- 完成度
- 频次和周期
- 增长或下降范围

除非用户要求，不要单独创建“数值汇总”章节。

## 改善、建议和计划

原文出现以下内容时要重点记录：

- 改善方向
- 问题修正
- 建议
- 下一步计划
- 待办事项
- 责任分工
- 后续跟进事项
- 明确时间节点

如果转写稿没有明确提到下一步计划或后续安排，不要自行编造该章节。

## 建议 Markdown 结构

可根据内容灵活使用以下结构：

```markdown
# <标题>｜内容萃取

## 基本信息

- 来源：
- 资料依据：
- 内容类型判断：

## 核心主线

...

## 重点内容

...

## 问题与原因

...

## 改善建议与后续安排

...

## 核心结论

...
```

只保留转写稿中有依据的章节。
